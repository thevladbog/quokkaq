# Яндекс.Трекер

**Яндекс.Трекер** — система управления задачами Яндекса. QuokkaQ может создавать задачи через REST API v3 и подтягивать статус в список обращений в приложении.

Официальная справка по API: [API Яндекс.Трекера](https://yandex.ru/support/tracker/ru/api-ref/).

## Поведение в QuokkaQ

- Пользователь отправляет **обращение** из интерфейса staff.
- Backend создаёт **задачу** в указанной очереди и сохраняет ключ задачи в PostgreSQL.
- Список в приложении читается **из API QuokkaQ**, а не напрямую из Трекера.
- Обращения **не удаляются** в Трекере; отметка «неактуально» добавляет **комментарий** к задаче и сохраняет `markedIrrelevantAt` локально.

## Локальные поля задачи при создании

В очереди Трекера заведите **локальные поля** с теми же идентификаторами, что и в JSON создания задачи. QuokkaQ заполняет:

| id поля                | Значение                                                                                                                                                                                                                                                                                                                                                                        |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `apiAccessToTheTicket` | Список **UUID пользователей QuokkaQ**, разделённый **запятыми** (CSV), с доступом к чтению этого обращения по API — те же правила, что и `ApiAccessToTicket` в бэкенде: автор, роль `admin` тенанта и пользователи с **расшариванием** в QuokkaQ. Значение пересчитывается при добавлении/снятии шаринга (PATCH с оптимистичным `version` в query, если Трекер его возвращает). |
| `applicantsEmailApi`   | Email автора обращения (`users.email`).                                                                                                                                                                                                                                                                                                                                         |
| `company`              | `<Название тенанта> (<краткое юр. наименование>)` — `companies.name` и при наличии `companies.counterparty.shortName` (иначе `fullName`) по компании автора (первая запись `user_units` → `units.company_id`). Если в counterparty нет краткого/полного наименования, в поле попадает только название тенанта (удобно для фильтра по тенанту).                                  |

## Выбор бэкенда

На сервере QuokkaQ API:

| Переменная                | Описание                                                                             |
| ------------------------- | ------------------------------------------------------------------------------------ |
| `SUPPORT_REPORT_PLATFORM` | `yandex_tracker` — Трекер, `plane` — Plane, не задано/другое — интеграция выключена. |
| `PLANE_ENABLED`           | При работе с Трекером задайте `false`, чтобы Plane не использовался по ошибке.       |

## Сеть и безопасность

- API Трекера должен быть доступен **с сервера QuokkaQ** (HTTPS).
- При IAM по ключу СА серверу также нужен доступ к **IAM Yandex Cloud** (`iam.api.cloud.yandex.net`) и к хосту **service discovery**, который использует SDK (HTTPS).
- Файл ключа храните **вне репозитория** (например, смонтированный секрет); не коммитьте JSON ключи. Каталог `temp/` в репозитории в `.gitignore`, но для прод-секретов он не подходит.

## Интерфейс staff: две колонки, комментарии, шаринг

- Карточка обращения в staff — **двухколоночный макет**: слева сохранённый текст обращения и **таймлайн комментариев**; справа техблок и **шаринг** (комментарии и шаринг только для Yandex Tracker; для Plane поведение прежнее).
- **Комментарии** читаются из Трекера (`GET /v3/issues/{id}/comments`) и классифицируются в API как `internal`, `public` или `email` (см. ниже). Отправка из QuokkaQ — обычный текст; **публичность для заявителя задаётся в Трекере**, а не переключателем в интерфейсе staff.

## Маркеры `[public]` и `[email]`

- **`public`**: после `TrimSpace` текст начинается с `[public]`; тег сравнивается **без учёта регистра**. В API для заявителя в `displayText` префикс убирается.
- **`email`**: если в JSON комментария Трекера есть признак почты (`type` и/или поле **`transport`** — часто `email`, `incoming`, **`outcoming`**, `outgoing`), комментарий считается перепиской. Тело письма может приходить в **`textHtml`**, пока `text` пустой; QuokkaQ запрашивает комментарии с `expand=all` и преобразует HTML в plain text для таймлайна. Иначе — префикс `[email]` по тем же правилам, что и `[public]`.
- **`internal`**: остальное (внутренняя переписка staff).

## API комментариев: заявитель и staff

- `GET /support/reports/{id}/comments?audience=staff` — полный таймлайн для тех, кто видит обращение (автор, админ тенанта или пользователь с **шарингом**).
- `GET /support/reports/{id}/comments?audience=applicant` — **только автор обращения**; в ответе только `kind` `public` и `email`, в `displayText` без служебных префиксов.

## Шаринг (ACL в QuokkaQ + синхронизация с Трекером)

- **Автор или админ тенанта** может искать коллег в **компании автора** (роли `admin`, `staff`, `supervisor`, `operator`) и выдавать доступ на чтение. Записи хранятся в PostgreSQL и попадают в `apiAccessToTheTicket` после каждого изменения.
- HTTP-маршруты — в сгенерированном OpenAPI / Swagger (раздел support): кандидаты, список, добавление и снятие шаринга.
