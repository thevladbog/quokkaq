# Developer API, вебхуки и публичный виджет очереди

Здесь — обзор для администратора тенанта. **Точные пути, тела запросов и схемы ответов** для вашего стенда смотрите в **OpenAPI**, который отдаёт ваш API (например Scalar по адресу `/swagger/`, JSON — `/docs/openapi.json` на хосте API).

## Возможности тарифа (`planCapabilities`)

В ответе **`GET /companies/me`** (авторизованный администратор тенанта) есть объект **`planCapabilities`** — набор флагов из подписки компании (у SaaS-оператора все функции могут быть включены независимо от тарифа):

| Поле | Смысл |
|--------|--------|
| `apiAccess` | REST интеграций по префиксу **`/integrations/v1`** (ключи API, scope на ключе). |
| `outboundWebhooks` | Настройка HTTPS endpoint’ов и подписанные события по жизненному циклу талонов. |
| `publicQueueWidget` | Встраивание read-only статуса очереди на свой сайт (allowlist `Origin` + короткоживущий JWT). |
| `kioskEmployeeIdp` | **Идентификация сотрудника (бейдж / логин, серверный прокси)** — внешний HTTPS URL и шаблоны в настройках подразделения, киоск вызывает **`POST /units/{unitId}/employee-idp/resolve`**; секреты не попадают в браузер. |

В том же ответе есть **`publicApiUrl`** — используйте его как базовый URL API в примерах кода, не хардкодьте домен из документации.

Если релевантные для вас флаги `false`, в интерфейсе обычно скрываются разделы Developer API / интеграций до смены тарифа.

## Employee IdP (бейдж / логин) — для **вашего HTTP API** каталога сотрудников (не REST QuokkaQ)

**Для кого:** команда, которая ведёт **каталог сотрудников / карт доступа** и отвечает на вопрос *«кому принадлежит этот бейдж / этот логин?»* для заданного подразделения.  
QuokkaQ **не передаёт** в браузер ключи к вашему API. Вызов идёт **только с бэкенда QuokkaQ** на **HTTPS** URL, который задаётся в настройках **юнита**. **Требуется** флаг `kioskEmployeeIdp` в **`planCapabilities`**; на киоске вызов — с **терминальной** (kiosk) сессией.

**Что внедрять у вас (HTTPS endpoint):**

- **URL** — один `https` URL на подразделение. Допустимы только **публичные** цели: loopback, «чистый» private IP в URL и **имена, резолвящиеся** только в частные/loopback сети, **запрещены** (защита от SSRF на стороне QuokkaQ).
- **Запрос:** тело — из **шаблона JSON** (Go `text/template`) в QuokkaQ, данные: **`Raw`**, **`Login`**, **`Kind`** (`badge` или `login`), **`Ts`**. Считыватель / wedge передают строку в `Raw` для `badge`; ввод с экрана — `kind: login` и тот же текст в **`Raw`/`Login`**.
- **Заголовки:** по желанию; в значениях — подстановка секретов, сохранённых в QuokkaQ, в виде **`${secret:ИМЯ}`**.
- **Ответ:** JSON. Email сотрудника читается по **пути gjson** (пример: `data.email`); по желанию — путь к отображаемому имени. Сопоставление с пользователем **в той же компании** по **email**; «одноразовая» выгрузка справочника **не** выполняется.

**Рекомендация по логам с вашей стороны:** не писать в чистом виде сырой бейдж/логин и не хранить полные ответы, если на это нет согласий; достаточно **идентификатора запроса** и **итога** (успех / не найден / ошибка) — в духе ваших обязательств 152‑ФЗ / DPA.

**Маршруты QuokkaQ (с вашей стороны вы их вызываете, это не тело HR API):**

- **`POST /units/{unitId}/employee-idp/resolve`**, тело `{ "kind": "badge"|"login", "raw": "…" }` (JWT терминала киоска). Контракт **HTTP-кодов** и полей `matchStatus` / `userId` — в **OpenAPI**.
- **Админка:** `GET`/`PATCH /units/{unitId}/employee-idp` при праве **`unit.employee_idp.manage`**. В `GET` **нет** значений секретов. В **`PATCH`** можно передать **`secretNamesToDelete`** — массив **имён** именованных секретов, которые нужно снять с подразделения.
- **Веб (сотрудники) и бейдж:** в **браузере** для сотрудников и админов используется обычный **SSO** продукта. **Resolve** по бейджу/логину (employee IdP) — это сценарий **киоска/терминала** с токеном, см. путь выше; не дублируйте resolve из «голого» браузера вместо SSO. **Бейдж** относится к **киоску** (и при необходимости к коллеру), не к веб-кабинету.
- **`matchStatus`:** кроме **`matched`** (с `userId`) и **`no_user`**, API может вернуть **`ambiguous`**, если e-mail с IdP совпал **более чем с одной** учётной записью в компании (тенанту надо убрать дубликаты). Маршрут **лимитируется** по IP и `unitId` (см. runbook; при превышении **HTTP 429**).
- **Режим услуги `identificationMode: "qr"`** на киоске требует включённой **предзаписи/чек-ина**; иначе не оставляйте пользователя без пояснения (см. in-app help и матрицу режимов).

## REST интеграций (`/integrations/v1`)

- Авторизация — **ключ API интеграции**, выпускается в настройках (имя заголовка и формат — в OpenAPI).
- У ключа есть **scopes**; у каждого маршрута в спецификации указаны нужные scope. Пример read-only: **`GET /integrations/v1/units/{unitId}/queue-summary`** (нужен `tickets:read` на ключе; для создания/изменения данных — `tickets:write`, как в OpenAPI).
- **Лимиты на `/integrations/v1` (на ключ):** при **`INTEGRATION_API_RL_REDIS=true`** и рабочем Redis — **до 180 запросов с одного ключа в скользящем окне 60 секунд**; счётчик общий для реплик. Иначе — in-memory: порядка **2 запр./с** устойчиво, **burst 60**; **лимитер сбрасывается при перезапуске** процесса API и **не шарится** между инстансами. См. флаг `INTEGRATION_API_RL_REDIS` и маршруты в OpenAPI.

## Вебхуки

- Создание, список, правка и удаление endpoint’ов — под **`/companies/me/webhook-endpoints`** (сессия пользователя с Bearer JWT и правами на настройки компании).
- **Ротация секрета подписи** — отдельный маршрут `POST …/rotate-secret` (см. OpenAPI); после ротации обновите проверку подписи на принимающей стороне.
- **Формат доставки, подписывающий заголовок, тело** — в **OpenAPI** (и Scalar) на вашем API; оно же — основной источник для админа тенанта. **Внутренние runbook’и оператора** (если у вас self-hosted и отдельная внутренняя дока) — только для инфраструктуры, не вместо OpenAPI.

## Публичный виджет очереди (embed)

**На стороне сервера** должен быть задан **`PUBLIC_WIDGET_JWT_SECRET`** — без него API не сможет выдавать и проверять HS256 JWT для встраивания. Токен подписывается и проверяется с этим секретом; поле **`exp`** в JWT задаёт срок жизни.

**Срок жизни JWT:** `POST /companies/me/public-widget-token` по умолчанию выдаёт токен на **900 секунд (15 мин.)**, если в теле не передан `ttlSeconds`. Можно задать `ttlSeconds` **от 60 до 86400** (1 сутки); сервер **не даст** TTL больше суток. Статус `publicApiUrl` и настройки компании смотрите в **`GET /companies/me`**. Смена **`publicQueueWidgetAllowedOrigins`** (в **`company.settings`**) влияет на CORS; при смене **`PUBLIC_WIDGET_JWT_SECRET`** **все** ранее выданные токены становятся недействительны — **перевыпустите** embed-скрипты. Обычно **обновляйте** токен **до истечения `exp`**, кэш браузера/страницы embed не «продлевает» срок.

**В браузере** запросы к виджету разрешены только с origin’ов из списка **`publicQueueWidgetAllowedOrigins`** в **`company.settings`**. Каждый элемент — полный origin вида `https://kiosk.example.com` (схема + хост, при необходимости порт; без пути). Настройка — в интерфейсе настроек вместе со сниппетом встраивания (в примере подставляется `publicApiUrl` из `GET /companies/me`).

**Эксплуатация:** смена `PUBLIC_WIDGET_JWT_SECRET` делает недействительными ранее выданные токены виджета, пока не перевыпустите embed.
