# SCIM 2.0 Enterprise Provisioning — Design Document

**Status:** Draft — all sections complete; awaiting user review before implementation plan.
**Date:** 2026-04-25
**Audience:** Backend / frontend / SRE
**Track:** Enterprise pack (sibling specs upcoming: custom domains, audit log streaming, public status page, IP allowlist, BYOK)
**Companion spec (parallel):** Health/readiness endpoints + error tracking (GlitchTip / self-hosted Sentry)

---

## Context

QuokkaQ is a multi-tenant SaaS with mature OIDC/SAML SSO (JIT provisioning) and RBAC. Russian enterprise / public-sector prospects routinely require SCIM 2.0 provisioning on procurement checklists — without it, deals do not progress to detailed evaluation.

This document specifies a v1 SCIM 2.0 implementation aligned with RFC 7643 / 7644, targeted at the Russian B2B / government market (152-ФЗ compliance, hosting in Yandex Cloud, integrations with КриптоПро, Avanpost FAM, Blitz Identity Provider, Keycloak).

**Out of scope for v1:** custom domains, BYOK encryption, IP allowlists, audit log streaming to customer SIEM, OAuth 2.0 client-credentials grant for SCIM. These are sibling Enterprise pack items, designed separately.

---

## Decisions (locked-in via brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | Reference SCIM 2.0 (RFC 7643/7644). IdP dialect adapters added later as middleware. | No pilot customer yet; broadest market coverage. |
| 2 | Lenient coexistence with existing JIT-SSO. Both work in parallel; matching by `userName` / `externalId`. | Backward compatible with current tenants; SCIM does not break existing onboarding. |
| 3 | Resource scope: **Users + Groups + tenant-configurable mapping table** (SCIM group → role + unit + services). | Covers Okta / Keycloak / Entra ID provisioning patterns. Manual-roles MVP would be too thin for enterprise. |
| 4 | Auth: **Per-tenant URL slug + Bearer token**. Token ↔ slug match enforced in middleware. | Explicit isolation in URL is preferred by RU sec-review teams; defence in depth vs. token compromise. |
| 5 | Deprovisioning: **soft-delete + retention-based PII anonymization** (152-ФЗ ст. 21 compliant). | Preserves audit / analytics history while honoring "right to erasure". |
| 6 | Architecture: **standalone `internal/scim/`** module, no external SCIM library. | Easier security review for RU enterprise; no third-party risk in auth-critical path. |

## Section 1 — Architecture & module layout

### 1.1 Module placement

Standalone `internal/scim/` package alongside existing `internal/auth/`, `internal/users/`, `internal/employee_idp/`, etc.

```
apps/backend/internal/scim/
├── handlers/
│   ├── users.go              # /Users — GET LIST/GET/POST/PUT/PATCH/DELETE
│   ├── groups.go             # /Groups
│   ├── service_provider.go   # /ServiceProviderConfig, /Schemas, /ResourceTypes
│   ├── bulk.go               # /Bulk (deferred to phase 2)
│   └── errors.go             # SCIM error envelope (status, scimType, detail)
├── service/
│   ├── user_service.go       # SCIM User ↔ domain User
│   ├── group_service.go      # SCIM Group + membership
│   ├── token_service.go      # gen / validate / rotate
│   └── mapping_service.go    # group → (role, unit, services) resolution
├── repository/
│   ├── user_repo.go          # filter / paginate / externalId lookup
│   ├── group_repo.go
│   ├── token_repo.go
│   └── mapping_repo.go
├── filter/
│   ├── lexer.go              # SCIM filter tokens
│   ├── parser.go             # AST
│   └── translator.go         # AST → GORM where-clauses
├── schemas/
│   ├── core.go               # urn:ietf:params:scim:schemas:core:2.0:*
│   ├── enterprise_user.go    # urn:ietf:params:scim:schemas:extension:enterprise:2.0:User
│   └── service_provider.go   # capabilities advertisement
├── middleware.go             # bearer token + slug check + rate limit + audit context
└── config.go                 # supported filters / sorts, rate limits, feature flags
```

### 1.3 Integration points with existing modules

| Existing | Change |
|---|---|
| `internal/users/service.go` | Add `CreateFromSCIM`, `UpdateFromSCIM`, `DeactivateFromSCIM`. They wrap existing `Create / Update / Deactivate`, passing `source=scim`, `external_id`, `scim_metadata`. |
| `internal/auth/` (SSO/JIT) | On first SSO login, look up user by SSO `externalId` claim against `users.scim_external_id`. If found → link, do not create. If not found → JIT creates as today (lenient mode). |
| `internal/audit_log_repo` | New `actor_type=scim`, `actor_id=<token_id>`. New event types: `scim.user.created/updated/deactivated`, `scim.group.created/updated/deleted/membership_changed`, `scim.token.generated/rotated/revoked`. |
| `internal/asynq` | New job `pii.anonymize_user` (scheduled per retention config). New scheduled job `scim.cleanup_old_audit` (housekeeping). |
| `cmd/api/main.go` | Register SCIM routes; register Asynq tasks; wire scheduled jobs. |
| `apps/backend/configs/openapi/` | SCIM endpoints **excluded** from main OpenAPI (separate standard). Generate optional `openapi-scim.yaml` for documentation only. |
| `apps/frontend` | New pages under `/settings/sso/scim/*` (enable, token management, group → role mapping, provisioning event log). |

### 1.4 Isolation & reversibility

- Global feature flag `SCIM_ENABLED=false` (env var) — middleware short-circuits with 404 on all SCIM routes.
- Per-tenant gating via existing subscription plan feature `scim_provisioning` (re-uses existing plan-based feature gates infrastructure).
- Rolling out / rolling back is a config toggle; no data migration required to disable.

## Section 2 — Data model & schema migrations

### 2.1 New tables (4)

#### `scim_provisioning_tokens` — bearer tokens for SCIM endpoints

```sql
CREATE TABLE scim_provisioning_tokens (
    id            UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id    UUID        NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
    name          VARCHAR(120) NOT NULL,             -- admin label, e.g. "Keycloak prod"
    token_hash    BYTEA       NOT NULL UNIQUE,       -- SHA-256(raw token)
    token_prefix  VARCHAR(16) NOT NULL,              -- first chars for display: "scim_a1b2"
    status        VARCHAR(16) NOT NULL DEFAULT 'active', -- active | revoked
    rotated_from  UUID        REFERENCES scim_provisioning_tokens(id),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by    UUID        REFERENCES users(id),
    last_used_at  TIMESTAMPTZ,
    last_used_ip  INET,
    revoked_at    TIMESTAMPTZ,
    revoked_by    UUID        REFERENCES users(id)
);

CREATE INDEX idx_scim_tokens_company_status ON scim_provisioning_tokens(company_id, status);
```

**Token format:** `scim_<prefix8>_<secret_b32_32>`. The DB stores only `SHA-256(raw)` plus the prefix. The full raw token is shown to the admin **once** at creation. We do not symmetric-encrypt it — since only the hash is stored, a DB leak does not allow token recovery.

#### `scim_groups` — SCIM Group resource

```sql
CREATE TABLE scim_groups (
    id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id       UUID        NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
    external_id      VARCHAR(255),                       -- IdP's group ID (recommended, optional per RFC)
    display_name     VARCHAR(255) NOT NULL,
    meta_created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    meta_updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at       TIMESTAMPTZ                         -- SCIM Group soft-delete
);

CREATE UNIQUE INDEX uq_scim_groups_company_externalid
    ON scim_groups(company_id, external_id)
    WHERE external_id IS NOT NULL AND deleted_at IS NULL;

CREATE UNIQUE INDEX uq_scim_groups_company_displayname
    ON scim_groups(company_id, display_name)
    WHERE deleted_at IS NULL;
```

#### `scim_user_groups` — m:n membership

```sql
CREATE TABLE scim_user_groups (
    user_id     UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    group_id    UUID        NOT NULL REFERENCES scim_groups(id) ON DELETE CASCADE,
    company_id  UUID        NOT NULL,                  -- denormalized
    added_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, group_id)
);

CREATE INDEX idx_scim_user_groups_company ON scim_user_groups(company_id);
CREATE INDEX idx_scim_user_groups_group   ON scim_user_groups(group_id);
```

#### `scim_group_mappings` — tenant config: group → grants

```sql
CREATE TABLE scim_group_mappings (
    id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id      UUID        NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
    scim_group_id   UUID        NOT NULL REFERENCES scim_groups(id) ON DELETE CASCADE,
    role_code       VARCHAR(64) NOT NULL,                  -- 'tenant_admin' | 'unit_manager' | 'operator'
    unit_id         UUID        REFERENCES units(id) ON DELETE CASCADE,  -- NULL for tenant_admin
    service_ids     UUID[],                                -- NULL = all services in unit (operator)
    is_active       BOOLEAN     NOT NULL DEFAULT true,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by      UUID        REFERENCES users(id),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_scim_mappings_company_group ON scim_group_mappings(company_id, scim_group_id);
CREATE INDEX idx_scim_mappings_unit          ON scim_group_mappings(unit_id);

-- Validation (CHECK or in service layer):
-- role_code = 'tenant_admin'  → unit_id IS NULL, service_ids IS NULL
-- role_code = 'unit_manager'  → unit_id NOT NULL, service_ids IS NULL
-- role_code = 'operator'      → unit_id NOT NULL, service_ids optional
```

Multiple rows per group are allowed (e.g. group `bank.operators.MSK` maps to operator@MSK-001 + operator@MSK-002).

### 2.2 Operational table: `scim_request_log`

Separate from `audit_log` (which is immutable / compliance-grade). This one is a **debug log** of every SCIM request with bounded retention.

```sql
CREATE TABLE scim_request_log (
    id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id           UUID        NOT NULL,
    token_id             UUID,
    method               VARCHAR(10) NOT NULL,
    path                 VARCHAR(255) NOT NULL,
    resource_type        VARCHAR(32),         -- User | Group
    resource_id          UUID,
    status_code          INTEGER     NOT NULL,
    scim_type            VARCHAR(64),         -- 'uniqueness' | 'noTarget' | 'mutability' | …
    error_detail         TEXT,
    request_body_summary JSONB,               -- truncated/redacted, no secrets
    duration_ms          INTEGER,
    client_ip            INET,
    created_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_scim_request_log_company_created ON scim_request_log(company_id, created_at DESC);
CREATE INDEX idx_scim_request_log_resource       ON scim_request_log(resource_type, resource_id);
```

Retention: scheduled job purges entries older than 90 days (configurable). Used by the admin UI "provisioning events" section.

### 2.4 Reusing `audit_log` (no schema change required)

SCIM events are written to the existing `audit_log` with:
- `actor_type = 'scim'`
- `actor_id = <token_id>`
- `event_type ∈ { scim.user.created, scim.user.updated, scim.user.deactivated, scim.user.anonymized, scim.group.created, scim.group.updated, scim.group.deleted, scim.group.membership_changed, scim.token.generated, scim.token.rotated, scim.token.revoked, scim.mapping.changed }`

### 3.2 Authentication contract

```http
GET /scim/v2/acme-bank/Users HTTP/1.1
Host: api.quokkaq.ru
Authorization: Bearer scim_a1b2c3d4_AB7CDF...32chars
Accept: application/scim+json
```

Middleware steps:
1. Extract `{slug}` → look up `companies.slug` → `company_id`. If not found → `404`.
2. Extract `Authorization: Bearer <token>` → `SHA-256(token)` → look up `scim_provisioning_tokens` where `status='active'`. If not found → `401`.
3. Verify `token.company_id == company_id`. Mismatch → `403` (separate metric for alerting).
4. Async update of `last_used_at`, `last_used_ip` (queued, not in hot path).
5. Write to `scim_request_log` after handler completion.

Rate limit: dedicated bucket per token — 30 req/sec, burst 100. Excess → `429` with SCIM error envelope.

### 3.6 Filter / sort / pagination contract

**Filter** (RFC 7644 §3.4.2.2 — full set supported):
- Operators: `eq, ne, co, sw, ew, pr, gt, ge, lt, le`
- Logical: `and, or, not`, grouping `()`
- Complex value filter: `members[value eq "user-uuid"]`
- Path attributes (incl. URN-prefixed): `urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department eq "ops"`

Filterable fields: `id`, `userName`, `externalId`, `email`, `active`, `meta.created`, `meta.lastModified`, `displayName` (Group). Others → `400 Bad Request` with `scimType: invalidFilter`. (`meta.created` and `meta.lastModified` are required for delta-sync patterns used by Okta and Entra ID.)

**Sort:** `sortBy` + `sortOrder` (`ascending` | `descending`). Supported fields: `userName`, `displayName`, `meta.created`, `meta.lastModified`.

**Pagination:** SCIM standard — **1-indexed**:
```
GET /Users?startIndex=1&count=50
```
Response:
```json
{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:ListResponse"],
  "totalResults": 247,
  "startIndex": 1,
  "itemsPerPage": 50,
  "Resources": [ … ]
}
```
`count` capped at 200 (advertised in ServiceProviderConfig).

**Attribute selection:** `?attributes=userName,emails` or `?excludedAttributes=phoneNumbers`. Supported — without it, responses are heavy for large organizations.

### 3.7 PATCH semantics (RFC 7644 §3.5.2)

Three operation types:
```json
{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
  "Operations": [
    { "op": "replace", "path": "active", "value": false },
    { "op": "add",     "path": "emails", "value": [{ "value": "new@x.ru", "type": "work" }] },
    { "op": "remove",  "path": "members[value eq \"user-uuid-1\"]" }
  ]
}
```

Operations are applied atomically in one transaction. If any operation fails, the entire PATCH is rejected (`400`). RFC-compliant.

Note: `op: replace, path: members` for Group **fully reassigns** membership. Large IdPs (Okta) use `add/remove` per change instead of `replace members`. Both patterns are handled.

### 3.8 Error envelope (RFC 7644 §3.12)

```json
HTTP/1.1 409 Conflict
Content-Type: application/scim+json

{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:Error"],
  "status": "409",
  "scimType": "uniqueness",
  "detail": "User with userName 'ivanov.i@acme-bank.ru' already exists."
}
```

Standard `scimType` values returned:
- `invalidFilter` — bad filter syntax / unsupported field
- `invalidPath` — bad path in PATCH
- `noTarget` — PATCH path matched nothing
- `uniqueness` — unique-constraint violation (userName, externalId)
- `mutability` — attempt to modify read-only attribute (`id`, `meta.created`)
- `tooMany` — `count` > maxResults

Plus standard codes: `401`, `403`, `404`, `429`, `500`, `503`. Each error written to `scim_request_log` with detail.

### 3.9 Conditional requests (ETag, If-Match)

Each resource returns `ETag: W/"v<n>-<lastModified>"` and `meta.version`. Updates (`PUT` / `PATCH`) accept optional `If-Match` — mismatch → `412 Precondition Failed` (protects against rare concurrent updates from multiple IdP channels).

For v1 ETag is best-effort: always emitted, only validated when the IdP sends `If-Match`.

### 3.10 Bulk endpoint — deferred

`POST /Bulk` is mentioned in the RFC, but IdP support varies (Okta: yes; many: no). Deferred to phase 2. ServiceProviderConfig honestly advertises `bulk.supported = false`.

## Section 4 — Group → role / unit / service mapping

### 4.1 Authority model — Layered (chosen)

Three considered:

- **A. Authoritative** — SCIM owns user grants entirely; manual assignments forbidden or wiped on next sync. Loses ad-hoc admin grants.
- **B. Additive** — SCIM grants stack on top of manual ones. Ambiguous when admin manually grants the same role that a SCIM group provides.
- **C. Layered (chosen)** — two namespaces tagged by `source`:
  - `source = 'scim'` — managed only by SCIM recompute
  - `source = 'manual'` — managed only by admin
  - Effective grants = `UNION(scim_grants, manual_grants)`

**Implementation:** add column `source VARCHAR(16) NOT NULL DEFAULT 'manual'` to the existing role-assignments table (exact name to be verified during implementation — `user_role_assignments` / `tenant_user_roles` / similar). SCIM recompute touches only `WHERE source='scim'`.

### 4.2 Mapping table semantics

A single SCIM group may have **multiple** mapping rows (different roles, units, services).

Example: group `bank.regional.MSK`:

| role_code | unit_id | service_ids | Effect |
|---|---|---|---|
| `unit_manager` | MSK-001 | NULL | Manager at branch MSK-001 |
| `operator` | MSK-001 | `[passport_id, id_card]` | Operator of those two services at MSK-001 |
| `operator` | MSK-002 | NULL | Operator of all services at MSK-002 |

A user member of this group receives all three grants.

### 4.3 Effective-grants computation algorithm

For a given `user_id` (pseudocode):

```python
def recompute_scim_grants(user_id):
    # 1. All groups of the user
    groups = SELECT scim_group_id FROM scim_user_groups WHERE user_id = ?

    # 2. All active mappings of those groups
    mappings = SELECT * FROM scim_group_mappings
               WHERE scim_group_id IN (groups) AND is_active = TRUE

    # 3. Collapse by (role_code, unit_id), unioning service_ids
    desired = {}  # key: (role_code, unit_id), value: set of service_ids
    for m in mappings:
        key = (m.role_code, m.unit_id)
        if key in desired:
            desired[key] |= set(m.service_ids or [])
        else:
            desired[key] = set(m.service_ids or [])

    # 4. Conflict resolution within the same unit_id:
    #    tenant_admin overrides unit_manager and operator
    #    unit_manager overrides operator (for the same unit_id)
    desired = apply_role_precedence(desired)

    # 5. Diff against current SCIM grants
    current = SELECT * FROM user_role_assignments WHERE user_id = ? AND source = 'scim'
    to_add, to_remove, to_update = diff(current, desired)

    # 6. Atomic in one transaction
    BEGIN
        DELETE FROM user_role_assignments WHERE user_id = ? AND source = 'scim' AND key IN to_remove
        INSERT INTO user_role_assignments (..., source='scim') VALUES to_add
        UPDATE user_role_assignments SET service_ids = ... WHERE user_id = ? AND key IN to_update AND source = 'scim'
    COMMIT

    # 7. Audit
    audit_log.write('scim.user.grants_recomputed', user_id=user_id, diff={...})
```

**Conflict precedence:** `tenant_admin > unit_manager > operator`. If a user has both `tenant_admin` (no unit) and `operator` in some unit — they are effectively tenant_admin (covers everything), but both rows are stored in DB (audit trail and clean reversal on group removal).

### 4.4 Recomputation triggers

| SCIM event | Whom to recompute | Sync / Async |
|---|---|---|
| `POST /Users` | The new user (often groupless → empty result) | Sync |
| `PATCH /Users/{id}` (non-membership attrs) | None | — |
| `POST /Groups` | None — empty group | — |
| `PATCH /Groups/{id}` add member X | User X | Sync |
| `PATCH /Groups/{id}` remove member X | User X | Sync |
| `PATCH /Groups/{id}` replace members | All added + all removed | Async if >10 users |
| `PUT /Groups/{id}` | Same as replace | Async |
| `DELETE /Groups/{id}` | All ex-members | Async |
| Admin saves mapping | All members of the group | Async |
| Admin deletes mapping | All members of the group | Async |
| Admin deletes unit / service | All members of all groups whose mappings reference that unit/service | Async (with mapping invalidation) |

**Async** via Asynq job `scim.recompute_user_grants(user_id)`. Idempotent — repeated calls converge.

The sync path does not block the SCIM response longer than ~50 ms even on large tenants: one user, one SQL transaction, indexed lookups.

### 4.5 Edge cases

| Case | Behaviour |
|---|---|
| Group with no mapping | User belongs to it but receives no grants. Highlighted in admin UI as "needs configuration". No SCIM error. |
| Mapping → deleted unit | When `units.deleted_at` is set, a DB trigger / service marks `scim_group_mappings.is_active=false` for all mappings on that unit. UI shows "broken". |
| Mapping → deleted service | Service ID silently skipped in computation; warning logged; mapping marked "incomplete" in UI. |
| User with no groups AND no manual grants | Can authenticate (lenient mode) but has no permissions. User UI shows "no role — contact admin". Admin UI segments these in a "stranded users" report. |
| Impossible mapping (operator without unit) | Admin UI form validation blocks; DB CHECK constraint as defence-in-depth. If somehow saved, ignored at compute time with warning log. |
| Group `members` references non-existent user | `400 invalidValue` with detail in SCIM response. Implicit user creation is not allowed. |
| Concurrent membership PATCHes | Each opens its own transaction; recompute-job is deduplicated by user_id (newer jobs supersede older in the queue). |
| Admin changes mapping while IdP sync in flight | Both trigger recomputes; last completed wins (UPSERT semantics). Worst case: one extra recompute cycle. |

### 4.6 Bootstrap when SCIM is first enabled

1. Tenant enables SCIM in `/settings/sso/scim` → token generated.
2. Tenant configures SCIM on the IdP side → IdP begins pushing.
3. IdP creates Users and Groups → they appear in QuokkaQ.
4. Admin sees, in `/settings/sso/scim/mappings`, a list of groups marked "needs configuration".
5. Admin configures mapping per group → async recompute kicks in.
6. Existing manual grants are untouched (`source='manual'`); SCIM grants are added on top (`source='scim'`).
7. Optional: admin can "convert" manual → scim for users now covered by SCIM (separate action in UI, not automatic).

### 4.7 Admin UX brief (full UX in Section 7)

`/settings/sso/scim/mappings`:
- Top section "Needs configuration" — groups without mappings.
- Table: `displayName | mapping rows | members count | actions`.
- Add / Edit mapping in modal with form validation (role_code → unit_id / service_ids constraints).

## Section 5 — Lifecycle & coexistence

### 5.1 Create flow (`POST /Users`)

End-to-end:
1. Middleware (auth, slug, rate limit, audit) — passed.
2. Parse SCIM resource + schema validation.
3. Uniqueness check: `userName` (= email) and `externalId` per company.
4. SCIM → domain mapping (Section 3.3).
5. `users.Service.CreateFromSCIM(...)`:
   - `is_active = active` (default true)
   - `scim_external_id`, `scim_metadata = { source: 'scim', first_provisioned_at, scim_token_id }`
6. If `groups[]` present in payload → INSERT `scim_user_groups`, sync recompute.
7. Audit: `scim.user.created`.
8. Response: `201 Created` + `Location` + `ETag`.

#### Conflict with existing user by `userName`

Common case: tenant enables SCIM but already has users (manually created or via JIT-SSO). IdP POSTs with the same email.

Three options:

- **A. Strict** — always `409 uniqueness`. IdP must resolve manually.
- **B. Auto-link always** — if existing user has `scim_external_id IS NULL` → link, set `scim_external_id` from request, update attributes, return `201`.
- **C. Auto-link as opt-in (chosen)** — per-tenant flag `scim.auto_link_existing_users` (default `false`). When `true` — behaviour B. When `false` — behaviour A. Audit event `scim.user.linked_existing` records the linkage.

### 5.2 Update flow (`PUT` vs `PATCH`)

**PUT** (full replace):
- All SCIM-mappable attributes are replaced.
- Missing → reset to default / null per RFC.
- Internal-only fields (`created_by`, `last_login_at`, custom QuokkaQ-specific fields) untouched.

**PATCH** (partial):
- Operations applied atomically in one transaction.
- Any failure → full rollback.

#### SCIM-managed vs admin-managed fields

For SCIM-provisioned users (`scim_metadata.source = 'scim'`), all SCIM-mappable attributes are flagged SCIM-managed:

| Attribute | SCIM-managed | Admin can edit |
|---|---|---|
| email / userName | ✅ | ❌ readonly in admin UI |
| first / last / middle_name | ✅ | ❌ |
| phone | ✅ | ❌ |
| is_active | ✅ | ⚠️ allowed but warns "may be overwritten by IdP" |
| locale, timezone, preferredLanguage | ✅ | ❌ |
| employee_id, department, manager_user_id | ✅ | ❌ |
| custom QuokkaQ fields (notes, tags, …) | ❌ | ✅ |
| `source='manual'` grants | ❌ | ✅ |

Admin UI displays a "Managed by IdP (SCIM)" badge next to readonly fields. Click → modal with instruction "change the value in your IdP".

JIT-provisioned users (`source='jit_sso'`) have editable attributes (current behaviour), until SCIM picks them up.

### 5.3 Deactivation (`PATCH active=false` / `DELETE`)

Both paths converge to the same outcome:
- `users.is_active = false`, `users.deactivated_at = now()`.
- All active sessions (JWT) invalidated via `user.deactivated` event.
- WebSocket connections force-closed for that user.
- Grants (SCIM and manual) remain in DB but enforcement returns nothing for inactive users.
- Audit: `scim.user.deactivated`.

`DELETE` returns `204 No Content` (per RFC). `PATCH active=false` returns `200 OK` with the updated User resource.

### 5.4 PII anonymization (retention)

Trigger: `deactivated_at + retention_period`. Default `365 days`, configurable per plan: `90 / 180 / 365 / never`.

Asynq scheduled job `pii.anonymize_users_due` (daily):
- SELECT users WHERE `deactivated_at < now() - retention` AND `pii_anonymized_at IS NULL`.
- For each — enqueue `pii.anonymize_user(user_id)`, atomic:
  ```
  email             → anon-{shortid}@deleted.local
  first_name        → NULL
  last_name         → NULL
  middle_name       → NULL
  phone             → NULL
  employee_id       → NULL
  scim_external_id  → NULL  (frees the ID for a future user)
  scim_metadata     → { source: 'scim', anonymized_at: now() }
  pii_anonymized_at = now()
  ```
- Audit: `scim.user.anonymized` with hashed PII summary (`email_hash`, `name_hash`) for forensics, no plaintext.

**Preserved:** `user_id` (audit-log integrity), `is_active=false`, `deactivated_at`, references in `audit_log` / `tickets` / `surveys` (user_id stays; data is gone).

If plan = `never` (for government tenants with explicit retention requirements) — anonymization is not auto-scheduled; remains an on-demand admin action.

### 5.5 Reactivation

- Inactive user, IdP sends PATCH `active=true`:
  - `is_active=true`, `deactivated_at=NULL`.
  - Recompute grants (mappings may have changed).
  - Audit: `scim.user.reactivated`.
- If `pii_anonymized_at IS NOT NULL` → `410 Gone` with detail "User anonymized; create a new resource".

### 5.6 JIT-SSO coexistence (lenient mode)

On SSO login (OIDC / SAML callback):

```
1. Look up by SSO subject claim → if user.scim_external_id = subject → found
2. Otherwise look up by email → if found, link sub into scim_external_id (when NULL)
3. If not found → JIT-create user (current behaviour), scim_metadata.source = 'jit_sso'
4. Authenticate
```

Per-tenant flag `scim.allow_jit_sso` (default `true`). If `false` — step 3 is disabled; unknown users get `403 Not provisioned, contact admin`. (Option A from brainstorm, kept as a future flag.)

When SCIM later POSTs the same user:
- `auto_link_existing_users = true` → links (see 5.1)
- `auto_link_existing_users = false` → `409`

### 5.7 Group lifecycle

| Action | DB effect | Recompute |
|---|---|---|
| `POST /Groups` | INSERT scim_groups (+ optional scim_user_groups if members) | Sync per added user |
| `PATCH /Groups` add member | INSERT scim_user_groups | Sync for added |
| `PATCH /Groups` remove member | DELETE scim_user_groups | Sync for removed |
| `PATCH /Groups` replace members | DELETE all + INSERT new | Async (if >10) |
| `PATCH /Groups` displayName | UPDATE | — |
| `PUT /Groups` | full replace | Async |
| `DELETE /Groups` | `scim_groups.deleted_at` set; `scim_group_mappings.is_active=false` | Async for all ex-members |

### 5.8 Admin actions vs SCIM

| Admin does | SCIM reaction |
|---|---|
| Deactivates a SCIM-managed user | Applied locally; on next IdP sync may be reversed to `active=true`. UI warns. |
| Deletes a SCIM-managed user | Soft-deletes; SCIM GET → `404` |
| Edits a SCIM-managed attribute | Blocked in UI; readonly badge |

### 5.9 Edge cases

| Case | Behaviour |
|---|---|
| `active=true` for anonymized user | `410 Gone` |
| PATCH `userName` to one already taken | `409 uniqueness` |
| DELETE already-deactivated user | `204` (idempotent) |
| DELETE non-existent user | `404` |
| POST with `userName` matching deactivated, not-yet-anonymized user | `409 uniqueness` with detail "reactivate or wait for anonymization" |
| Group with 10000 members; admin saves mapping | Async job, batched at 100/sec via Asynq rate limit, to avoid DB spike |

## Section 6 — Security, audit, observability

### 6.1 Token security

**Generation:**
- Format: `scim_<8-char prefix>_<32-byte secret base32-encoded>` ≈ 256 bits of entropy.
- Server-side, **shown to admin once** at creation.
- No symmetric encryption — only `SHA-256(token)` is stored, so a DB leak does not allow recovery.
- Hash algorithm: SHA-256 (sufficient since the input already has 256 bits of crypto-random entropy).
- Validation: constant-time hash comparison + indexed lookup.

**Rotation:**
- Admin clicks "Rotate" → new token, row created with `rotated_from=<old_id>`.
- Old token marked `revoked` after a grace period (default 24h, configurable).
- During grace period **both tokens are accepted** — IdP can rotate without downtime.
- Audit: `scim.token.rotated`.

**Revocation:** immediate (`status=revoked`, `revoked_at`). Audit: `scim.token.revoked`.

### 6.2 Rate limiting (3 layers)

| Layer | Limit | Storage |
|---|---|---|
| Per-token | 30 req/sec, burst 100 | Redis bucket, TTL 1h, key=token_id |
| Per-token write | 10 req/sec, burst 50 for `POST /Users`, `PATCH /Groups` | Redis bucket, key=token_id+endpoint |
| Per-IP general | existing public-rate-limit middleware | unchanged |

Excess → `429` with SCIM error envelope + `Retry-After`.

Anti-DoS measures on the filter parser:
- 5s parsing timeout
- Max AST depth 32
- `scim_filter_complexity_seconds` histogram for outlier detection

### 6.3 Audit log — event catalogue

Written to existing `audit_log` with `actor_type='scim'`, `actor_id=<token_id>`:

```

scim.user.updated
scim.user.deactivated
scim.user.anonymized
scim.user.reactivated

scim.token.rotated
scim.token.revoked

```

Each entry contains: `user_id` or `group_id`, summary diff (what changed), client IP, request_id (for trace correlation).

`audit_log` is immutable, indexed `(company_id, event_type, created_at)`. UI access — see Section 7.

```

```

Async recompute-job spans are linked via `parent_span_id` for end-to-end tracing of "SCIM PATCH → recompute → audit log".

### 6.6 Structured logging (slog) with PII redaction

```go
slog.Info("scim.request",
    "tenant_id", tenantID,
    "token_id", tokenID,
    "method", "POST",
    "path", "/Users",
    "resource_id", userID,
    "status_code", 201,
    "duration_ms", 23,
    "client_ip", clientIP,
    "scim_type", scimType, // only on error
)
```

**PII redaction is mandatory:**
- Logs **never contain** plain emails, names, phone numbers.
- For correlation use `email_hash` (SHA-256, first 12 chars).
- `scim_request_log.request_body_summary` (JSONB) — schema keys only, values either redacted or stripped for PII fields.

Linter rule: PR pre-commit hook greps for `slog\.(Info|Error).*"email"` and similar PII keywords.

### 6.7 Threat model

| Threat | Mitigation |
|---|---|
| Token compromise | Hash storage; immediate revoke; rotation; (optional, phase 2) IP allowlist; alert on `scim_token_misuse_total` rate |
| Slug enumeration | Unified `401` on auth failure (no leak of "tenant exists vs token invalid") |
| Replay | TLS mandatory; idempotent ops; ETag / If-Match for protected updates |
| IDOR (cross-tenant data) | All queries scoped by `company_id` from middleware; covered by integration tests with two tenants |
| SQL injection via filter | Filter parser → AST → parametrized GORM queries; AST cannot emit raw SQL |
| Mass enumeration via `GET /Users` | 30 rps rate limit; admin alert if >10 000 req/hour; pagination max 200 |
| DoS via complex filters | 5s timeout + max depth 32 + complexity histogram |
| Insider abuse (admin issues a token, exfiltrates data) | All token actions audited; cron `scim.token.usage_anomaly` alerts on suspicious patterns (sudden burst, atypical IP, off-hours) |

**152-ФЗ:**
- ст. 9 (consent): out of SCIM scope — tenant (as data operator) has obtained consent.
- ст. 18.1 (DPA): QuokkaQ acts as "лицо, осуществляющее обработку"; tenant is "оператор".
- ст. 21 (destruction): retention + anonymization (Section 5.4).
- ст. 22.1 (processing log): `scim_request_log` + `audit_log`.

**SOC 2 prep:**

- The SOC 2 audit itself is a separate compliance project; this design is compatible.

### 6.9 Secrets management

- `SSO_SECRETS_ENCRYPTION_KEY` (already in env) is **not used** for SCIM tokens — only the hash is stored.

- Token rotation = DB write only, no external vault calls.

## Section 8 — Testing strategy & rollout

### 8.1 Test pyramid

**Unit (Go):**
- Filter parser / AST: 50+ cases covering all operators + edge cases + error paths.
- SCIM ↔ domain mapping: round-trip every attribute.
- Conflict resolution algorithm: precedence, multiple groups, conflicting roles.
- Token validation: hash comparison, slug-token binding, revoked / expired status.
- Recompute algorithm: idempotency, correct diffs.
- Anonymization function: every PII field cleared.

**Integration (Go + real Postgres + Redis):**
- Full flows: `POST User` → `PATCH Group` add → recompute → `audit_log` entry.
- Multi-tenant isolation: 2 company fixtures; cross-slug requests → 403 (regression test against IDOR).
- Token lifecycle: generate → rotate → grace period → revoke.
- Lenient JIT-SSO: SSO finds SCIM user; SSO creates JIT user; later SCIM auto-links.
- Retention + anonymization: clock manipulation simulates timer.
- All `scimType` errors return correct envelope.

**E2E (Playwright — coordinated with the parallel ops spec):**
- Admin UI: enable SCIM, generate token (single-shot reveal), copy-to-clipboard.
- Mappings: add mapping → "needs configuration" disappears → recompute completes.
- Tokens: rotate → both tokens accept during grace → old revoked.
- Activity feed: filter by event type.

### 8.2 RFC conformance — external validators

A dedicated CI job `scim-conformance` runs the backend against:
- **Microsoft Entra ID SCIM Validator** (free online tool; can be self-hosted or mocked)
- **Okta SCIM Tester** (via Okta developer account)
- **RFC 7644 reference test cases** (where available)

Goal: catch dialect surprises before a pilot customer hits them.

### 8.3 Test fixtures

Corpus of real payloads from major IdPs (with redacted tokens):
- Okta: User Create / Update / Deactivate, Group Create / member add/remove
- Keycloak: same scenarios
- Microsoft Entra ID Application Provisioning
- (as pilots appear) Avanpost FAM, Blitz Identity

Stored at `apps/backend/internal/scim/testdata/payloads/<idp>/<scenario>.json`.

These fixtures double as regression tests for dialect adapters in phase 2.

### 8.4 Load testing (k6 — coordinated with ops spec)

| Scenario | Expected load | Target p95 / p99 |
|---|---|---|
| Burst `POST /Users` (10 000 users in IdP, new tenant) | 100 concurrent | 200 ms / 500 ms |
| Sustained `PATCH /Groups` membership churn | 500 concurrent | 150 ms / 400 ms |
| `GET /Users` paginated (large org) | 30 rps | 100 ms / 250 ms |
| Recompute storm (1 000 users → mapping changed) | sustained | throughput 100 users/sec |

Acceptance: no 5xx; no DB pool exhaustion; queue size does not grow linearly.

### 8.5 Rollout plan

Week numbers below are counted from the **implementation kickoff date** (T0), not calendar weeks. Cadence is the floor — phases can run longer if conformance / load tests surface issues.

**Phase 1 — Internal alpha (T0 + week 1–2):**
- `SCIM_ENABLED=true` on staging only.
- Internal QA tenant with self-hosted Keycloak + Okta dev account.
- Run conformance + integration suites.
- Verify metrics / traces in OTel pipeline.

**Phase 2 — Limited beta (T0 + week 3–4):**
- `SCIM_ENABLED=true` globally in prod.
- Per-tenant flag `scim_provisioning` enabled for 1–2 hand-picked beta tenants.
- Daily monitoring of `scim_token_misuse_total`, `scim_request_duration_seconds` p99.
- Direct support channel with beta tenants.

**Phase 3 — GA (T0 + week 5+):**
- Open `scim_provisioning` to plan tier ≥ Business.
- Public docs at `docs.quokkaq.ru/scim`.
- Marketing announcement.

**Rollback:**
- Global kill: env `SCIM_ENABLED=false` → all SCIM routes return `404`.
- Per-tenant kill: disable `scim_provisioning` plan feature.
- DB migrations are forward-only in normal flow; no data deletion required — tokens and mappings remain dormant until re-enabled.

### 8.6 Post-launch monitoring & alerts

| Alert | Threshold | Severity |
|---|---|---|
| `scim_token_misuse_total{type=slug_mismatch}` | >5/min per tenant | high (possible token leak) |
| `scim_request_duration_seconds` p99 | >1 s | medium |
| `scim_recompute_jobs_queue_size` | >1 000 | medium (async backlog) |
| Any 5xx on SCIM endpoints | any | high (page on-call) |
| `scim_token_misuse_total{type=revoked}` | >1 | medium (IdP did not update) |
