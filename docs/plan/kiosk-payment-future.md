# Оплата на киоске (посетитель) — отдельная инициатива (5.6)

**Статус:** **вне** текущей поставки фазы 5 (**5.1–5.5**). Требуется **самостоятельная** проработка: продукт, PCI, договоры, эквайринг, фискализация, развод с **SaaS-инвойсингом**. Этот документ — **заготовка** для **будущей** реализации, **не** часть плана `kiosk-phase5-delivery` и **не** чек-лист к текущему релизу.

**Связь:** [п. 5.6 в роадмапе](../plans/kiosk-roadmap.md).

**Зачем отдельно от 5.1–5.5:** интеграции YooKassa/Stripe в проекте сейчас — контур **подписки/инвойсов** платформы. Платёж **гостя** к **тенанту** — другой поток, webhooks, идемпотентность, юр./налог. Имеет смысл **не** смешивать с поставкой аналитики/ETA/опроса/OCR/offline. Подход к `plan features`: [`PlanFeature` / `CompanyHasPlanFeature`](../../apps/backend/internal/services/plan_feature.go); ключ для 5.6 — **`kiosk_visitor_payment`**, подключать при **запуске** этой инициативы.

## Целевой объём (для проработки, не дедлайн)

Ниже — **черновик** того, что **стоит** проектировать до разработки (без требования «сделать всё сразу в одной итерации» — **это** и есть предмет **отдельного** плана после обдумывания).

1. **Данные и конфиг:** `KioskConfig.visitorPayment`, при необходимости модель **`VisitorPayment`**, согласование с **уже** существующим 5.5 (например, не предлагать оплату в **offline**).
2. **API** создания сессии оплаты, **отдельные** webhook-маршруты (не invoice handlers), идемпотентность.
3. **Провайдеры:** по результатам анализа — Stripe и/или YooKassa на **аккаунт тенанта** vs **platform** (продукт+юристы).
4. **Kiosk / Tauri:** hosted-страница, return URL, при необходимости абстракция под **внешний** POS.
5. **Админка** и **sandbox.**
6. **Совместимость** с 5.5: правила, когда **запрещать** `visitorPayment`.
